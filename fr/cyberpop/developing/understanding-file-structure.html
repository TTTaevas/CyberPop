Bien comprendre la structure des fichiers de CyberPop

Cette leçon est là pour vous aider à comprendre comment fonctionne la structure des fichiers de CyberPop, ce qui devrait vous intéresser uniquement si vous êtes intéressé par développer CyberPop ou en faire votre propre version.

Si vous êtes toujours intéressé, bienvenue !
CyberPop est un site web qui n'utilise que des technologies de "front-end" (des fois traduit en "web frontal"), il utilise seulement du HTML, du CSS et du JS.
Les pages web, les polices d'écritues, les images, les scripts, tout ça, c'est localisé sur le même serveur, il n'y a aucune utilisation de serveurs tiers, mises à part pour toutes les images utilisées dans les "pages web d'exemple" (exemple CSS 2-2), qui sont localisées sur les serveurs de Wikimedia afin d'économiser de la mémoire.

Toutes les pages web de CyberPop sont pré-construites et sont composées de plusieurs parties séparées, qui sont chargées du côté du client par toute l'automatisation de laquelle nous parlons dans la prochaine leçon.
Ces parties diffèrent en fonction de quelle partie du site web vous êtes.

En gros, le site web est divisé par le nombre de langues disponibles. Ces parties sont representées par le code ISO 639-1 de leur langue respective, donc par exemple, la partie Anglaise du site web peut être retrouvée dans le dossier "en" pendant que la partie Française se retrouve dans le dossier "fr".

Dans le "root folder" (traduit parfois par "répertoire racine"), dans index.html se trouve l'écran où l'on choisit la langue. Il envoit l'utilisateur dans l'index.html du bon dossier, donc si l'utilisateur choisit "ENGLISH", il sera envoyé dans l'index.html du dossier "en", qui charge notamment l'entête et le bas de page du même dossier.

Mises à part les "pages web d'exemple", chaque page web dîte "visitable" qui n'est pas un enfant direct d'un dossier nommé après une langue utilise dans sa <head> un élément <base> faisant référence à ce dossier, donc quand elle charge quelque chose comme une entête, c'est relatif au dossier plutôt qu'eux-mêmes; la tête n'a pas a être modifiée pour chaque page, sauf pour ce même élément <base>.

Les scripts utilisés par le website tout entier se retrouvent dans le "root folder" pour rendre les choses simples, alors que le css et les images telles que les fonds peuvent être trouvés dans leur propre dossier, qui sera toujours localisé dans le "root folder".
Cependant, les images utilisées seulement par des leçons spécifiques sont localisées dans leur prore dossier qui est localisé dans le dossier de la catégorie de cette même leçon, donc par exemple, les images utilisées dans la leçon précédente peuvent être trouvées dans le dossier images qui se trouve dans le dossier "cyberpop".

Ce n'est pas exactement la même logique qui s'applique aux CSS&JS spécifiques aux "pages web d'exemple". Non seulement se trouvent-ils dans le même dossier que leur "page web d'exemple", mais ils partagent aussi le même nom, avec la différence de l'extension.
Une "page web d'exemple" aura une extension .html, alors que les éléments qu'elle utilise auront une extension .css, .js ou autre chose du genre.

Vous avez peut-être remarqué le fichier "404.html" qui peut être trouvé dans le "root folder".
Il fonctionne d'une façon complètement différente que les autres pages web dîtes "visitables" à cause du fait qu'elle ne marche exclusivement que sur "GitHub Pages" et à cause de comment ça fonctionne.

Quand hébergé sur ce service, l'URL du siteweb sera quelque chose comme "votrenom.github.io/CyberPop", ce qui peut faire que les pages web pensent que le "root folder" est le parent du vrai "root folder", car les sites web ont généralement une URL qui ne suit pas un slash ("/").
C'est la raison pour laquelle cette page web utilise un élément <base> malgré le fait qu'elle soit dans le "root folder" !

En espérant que ça couvre tout ce qu'il y a sur le sujet !
